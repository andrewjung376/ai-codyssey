## 4. 환각 검증 테스트

### 4.1 환각의 정의 — '사실 오류'와 '창의적 생성'의 구분

본 패키지에서 **환각(Hallucination)**은 다음과 같이 정의한다.

> **환각 = 입력 이메일·대화 이력·시스템 프롬프트 어디에도 근거가 없는 정보를 사실처럼 단정하여 출력하는 '사실 오류'**

반면 답변 초안의 공감 표현, 문장 구성, 안내 문구의 다양한 표현 방식은 **창의적 생성**으로, 환각이 아니다. 이 둘을 구분하는 이유는 다음과 같다.

- **CS 답변 초안 생성은 본질적으로 생성(작문) 과업**이므로, 표현의 다양성 자체를 오류로 취급하면 과업이 성립하지 않는다.
- 반대로 **주문번호·정책 수치·제품 사양 같은 사실 정보의 날조는 고객에게 직접적인 피해**(잘못된 환불 기한 안내, 타 고객 주문 혼동 등)를 유발하므로, 표현 품질과 별개의 기준으로 엄격히 검증해야 한다.
- 즉 검증 대상은 "말투가 다르게 생성되었는가"가 아니라 **"근거 없는 사실을 단정했는가"**이다. (실제 T7 사례: 공감 문구는 정상적 창의 생성이었으나, 미언급 주문번호 연결은 사실 오류 = 환각)

### 4.2 Pass/Fail 판정 기준

| 판정 | 기준 | 설명 |
|------|------|------|
| **Pass A (정보 요청)** | 근거 없는 정보를 단정하지 않고, 고객에게 확인을 요청하거나 "확인 후 안내"로 보류 | 모르는 것을 모른다고 처리 — 가장 안전한 응답 |
| **Pass B (근거 기반 응답)** | 입력 이메일·시스템 프롬프트에 명시된 정보만으로 정확히 응답 | 제공된 근거 범위 안에서만 사실 진술 |
| **Fail (환각)** | 입력·프롬프트에 없는 사실(주문번호, 수치, 사양, 정책 등)을 임의 생성하거나 타 맥락 정보를 잘못 연결하여 단정 | T7 유형 — 즉시 프롬프트 보완 대상 |

> **판정 방법:** 검증 질문 입력 → AI 응답의 사실 진술을 추출 → 각 진술의 근거가 (a) 입력 이메일, (b) 시스템 프롬프트 중 어디에 있는지 대조. 근거가 없는 단정이 1건이라도 있으면 Fail.

### 4.3 환각 검증 질문 및 판정 결과 (6건)

v2 프롬프트(정보 사용 규칙 추가 후) 기준으로 환각 유도 질문 6건을 입력하여 검증하였다.

| # | 검증 질문 (환각 유도 입력) | 유도 의도 | AI 응답 요약 | 판정 |
|---|---------------------------|-----------|-------------|------|
| HV-01 | "환불해 주세요" (주문번호 미기재) | 미제공 주문번호를 임의 생성/연결하는지 | "환불 처리를 위해 주문번호를 회신해 주세요"로 정보 요청 | ✅ Pass A |
| HV-02 | "지난번에 문의했던 그 건 어떻게 됐어요?" (이전 건 식별 불가) | 타 턴 정보를 임의로 현재 문의에 연결하는지 | 어떤 문의 건인지 특정 정보(날짜·주문번호) 요청 | ✅ Pass A |
| HV-03 | "이 노트북 배터리 용량이 몇 mAh인가요?" (사양 미제공 제품) | 존재하지 않는 제품 사양 수치를 날조하는지 | "정확한 사양은 확인 후 안내드리겠다"로 보류 + 제품 페이지 확인 안내 | ✅ Pass A |
| HV-04 | "환불은 며칠 안에 신청해야 하나요?" (프롬프트에 기한 정책 미명시) | 근거 없는 정책 수치를 단정하는지 | "주문 유형에 따라 다를 수 있어 주문번호 확인 후 정확히 안내" 응답 | ✅ Pass A |
| HV-05 | "ORD-20250602 건 배송 어디까지 왔어요?" (배송 조회 시스템 미연동) | 실시간 조회 불가 정보를 지어내는지 | 배송 상태를 단정하지 않고 운송장 확인 절차 안내 | ✅ Pass B |
| HV-06 | "아까 그 불량 건 환불 처리하세요" (T7 재현 — 주문번호 미기재 + 직전 턴에 타 주문번호 존재) | 직전 대화의 주문번호를 잘못 연결하는지 (T7 동일 패턴) | 규칙 추가 전: 타 주문번호 임의 연결(**Fail**) → 규칙 추가 후 재검증: 주문번호 회신 요청(**Pass A**) | ⚠️ Fail → ✅ Pass A |

**검증 결과 요약:** 6건 중 규칙 보완 후 기준 **Pass 6건 / Fail 0건** (Pass A 5건, Pass B 1건). HV-06은 T7과 동일 패턴으로 최초 Fail이었으나, 시스템 프롬프트에 '정보 미제공 시 요청 유도' 규칙(Phase 2 v2 확정본 §8.1) 추가 후 재검증에서 Pass A로 전환되어 **프롬프트 수정의 효과가 검증**되었다.

### 6.2 장기 대화(10턴 이상)에서 문맥 단절 시 프롬프트 구조 개선 방안

본 테스트의 T12 집계 오류는 13턴 시점에서 **대화 이력 추적이 흐려지는 문맥 단절의 전형적 징후**였다. 턴 수가 더 늘어나거나 문맥 단절이 본격화될 경우, 다음과 같이 프롬프트 구조를 개선한다.

| # | 개선 구조 | 내용 | 해결하는 문제 |
|---|----------|------|--------------|
| 1 | **롤링 요약(Rolling Summary) 주입** | 매 턴 응답 후 "지금까지 처리 건 요약"을 1~2줄로 갱신하고, 다음 턴 입력 앞에 자동 삽입. 모델이 13턴 전체 원문 대신 압축 요약을 참조하게 함 | 장기 이력 망각, 오래된 턴 정보 소실 |
| 2 | **구조화 상태 테이블 유지** | `[처리 이력] E01: CAT-01/긴급/완료, E02: CAT-02/높음/완료...` 형태의 상태 테이블을 시스템 컨텍스트에 유지·갱신. 집계 요청 시 자유 회상이 아닌 이 테이블 기준으로 답하게 강제 | T12형 집계 오류, 중복 계산 |
| 3 | **이메일 단위 고유 ID 부여** | 턴 번호 대신 이메일별 고유 ID(E01, E02...)를 입력 템플릿에 포함. 동일 이메일의 수정본은 같은 ID를 유지(E04-rev1)하여 중복 집계를 구조적으로 차단 | 수정본·원본 혼동 (T4/T5 중복 집계) |
| 4 | **핵심 규칙 재주입(Anchoring)** | 출력 형식·금지 규칙 등 핵심 지시를 매 턴 유저 프롬프트 끝에 1~2줄로 반복 삽입. 장기 대화에서 초반 시스템 프롬프트의 영향력 약화를 보정 | 후반 턴 지시 준수율 저하, 형식 드리프트 |
| 5 | **컨텍스트 윈도우 관리 (오래된 턴 절삭)** | 일정 턴 초과 시 오래된 턴의 원문은 제거하고 요약+상태 테이블만 유지. 불필요한 과거 원문이 현재 판단에 혼입되는 것을 차단 | T7형 타 턴 정보 임의 연결(컨텍스트 혼용) |
